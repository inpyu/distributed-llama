즉:
```
8B는 compute-bound가 아니라 communication-bound
```

# 4️⃣ 8B vs 3B 차이 분석

|항목|8B|3B|
|---|---|---|
|파라미터 크기|큼|작음|
|weight shard 크기|큼|작음|
|all-reduce 비용|매우 큼|중간|
|통신 비율|매우 높음|높음|
|scaling 효율|매우 나쁨|나쁨|

---

## 💡 핵심 차이

### 8B
- compute 대비 통신 비율 매우 큼
- workers 증가 → 통신이 지배
- scaling 거의 불가능

### 3B
- 계산량이 적음
- 통신 비율 상대적으로 낮음
- 하지만 여전히 network bound

# 6️⃣ 전체 결론

## 📌 1️⃣ scaling은 실패

두 모델 모두:

```
Workers ↑ → tokens/s ↓
```

즉 strong scaling 실패

---
## 📌 2️⃣ collective auto 효과

- 미세한 최적화
- TTFT 약간 개선
- 하지만 구조적 병목 해결 못함
    

---
## 📌 3️⃣ 병목 원인

1. Ring all-reduce 비용
2. 매 token마다 collective 발생
3. 네트워크 latency > compute 시간
4. 작은 batch (nTokens=2 prefill)
    

---
# 7️⃣ 연구 관점 해석 (랩미팅용 요약)

> dllama의 TP 기반 구조는 작은 모델에서도 통신 비용이 계산 이득을 상쇄하며,  
> 특히 8B 모델에서는 strong scaling이 완전히 무너진다.  
> collective auto는 통신 패턴을 약간 최적화하지만,  
> per-token all-reduce 구조 자체가 decode 단계에서 지배적 병목이 된다.

---
# 8️⃣ 향후 개선 방향 제안

1. Reduce-scatter + All-gather 구조 전환
2. Decode 단계 collective frequency 감소
3. Sequence parallel 적용 실험
4. Larger batch 실험
5. Star topology vs Ring 비교
6. KV-cache locality 개선

# 🔟 재배포 후 PP/Prefill 파라미터 재실험 (2026-03-03)

전제:

- 원격 8-worker 재배포 완료 후 재측정
- Model: `dllama_model_original_q40.m`
- Tokenizer: `dllama_tokenizer_llama3.t`
- Buffer: `q80`
- Threads: `4`
- Steps: `420`
- Collective: `auto`
- net-turbo: `0`

## 결과 요약

케이스 이름 규칙:

- `pXa`/`pXb`/`pXc`/`pXd`에서 `X`는 `pp-size`를 의미한다.
- 접미사(`a`,`b`,`c`,`d`)는 같은 `pp-size` 내에서 `prefill-chunk-threshold`/`prefill-chunk-size` 조합을 구분하기 위한 실험 인덱스다.

요청된 케이스 의미:

- `p1a` = `pp=1, threshold=128, chunk=0`
- `p1b` = `pp=1, threshold=64, chunk=0`
- `p2a` = `pp=2, threshold=64, chunk=0`
- `p2b` = `pp=2, threshold=64, chunk=8`
- `p2c` = `pp=2, threshold=128, chunk=0`
- `p2d` = `pp=2, threshold=128, chunk=8`

| Case | pp-size | threshold | chunk-size | Exit | Eval tok/s | Pred tok/s | TTFT(ms) | Decode(ms) | 비고 |
|---|---:|---:|---:|---:|---:|---:|---:|---:|---|
| p1a | 1 | 128 | 0 | 0 | 1.35 | 1.24 | 149276.83 | 180201.17 | 매우 느리지만 완료 |
| p1b | 1 | 64 | 0 | 1 | - | - | - | - | `SYNC_NODES` stall 후 `Socket closed` |
| p2a | 2 | 64 | 0 | 0 | 6.50 | 5.73 | 73962.23 | 82718.02 | 완료 |
| p2b | 2 | 64 | 8 | 1 | - | - | - | - | `SYNC_NODES` stall 후 `Socket closed` |
| p2c | 2 | 128 | 0 | 0 | 6.61 | 5.77 | 72642.38 | 82499.24 | 완료 |
| p2d | 2 | 128 | 8 | 1 | - | - | - | - | `SYNC_NODES` stall 후 `Socket closed` |
| p4a | 4 | 64 | 0 | 0 | 33.79 | 26.29 | 25066.87 | 28741.06 | 완료 |
| p4b | 4 | 64 | 4 | 0 | 33.32 | 26.37 | 24959.85 | 28985.66 | 완료 |
| p4c | 4 | 128 | 0 | 0 | 33.52 | 26.30 | 24981.24 | 28801.28 | 완료 |

## 해석

1. **현 시점 최적 구간은 `pp-size=4`**이며, `Pred tok/s`가 약 `26.3~26.4`로 가장 높고 안정적으로 완료된다.
2. `pp-size=2`는 `chunk-size=0`일 때는 안정적이지만, `chunk-size=8`을 주면 `step=16/227`에서 반복적으로 `SYNC_NODES` stall + `Socket closed`가 재현된다.
3. `pp-size=1`은 성능이 매우 낮고(`Pred 1.24 tok/s`), 특정 조건(`threshold=64`)에서 조기 stall이 발생한다.
